**Q3. Server-side mapping and policy**

- No GUID→profile map exists anywhere server-side today (`PlayerManager.h/.cpp` keys only by ephemeral `ConnectionId`; `PlayerComponent.h` has no ID field). This must be added: a `std::string Guid` (or fixed 16-byte array) field on `PlayerComponent`, plus a second index `Map<std::string /*guid*/, ...>` in `PlayerManager` (or a persisted-profile store keyed by GUID, which is what R5 presumably reads) — this is new code, not a rewire of existing storage.
- Persistence precedent for server-side per-identity state already exists: `code/server/scripting/JobSystem/WorldClock.cs:23,95-125` — plain `File.ReadAllText`/`WriteAllText` against a flat file (`world-clock.txt`) next to the server binary, loaded in the constructor, saved on a timer. The same pattern (one line/JSON entry per GUID, loaded at server start, appended/rewritten on join) is the minimal-effort persistence for a GUID→display-name profile table; no DB dependency needed for a friends-server scale.
- Uniqueness policy (design recommendation, not yet implemented anywhere): reject empty name (currently nothing rejects empty `username` — `GameServer.cpp:212` — this check must be added at `HandleAuthentication` before `PlayerManager::Create`); on collision, auto-suffix (`Name`, `Name (2)`, …) rather than hard-reject, since rejecting forces a friend to renegotiate a name mid-join with no UI feedback channel back to the client (the protocol has no "authentication error reason for retry" round-trip beyond `AuthenticationResponse.error` + immediate `Kick()`, `server.proto:11-15`, `GameServer.cpp:177-178` — a rejection today is a hard disconnect, not a retry prompt).
- Token-mismatch-for-existing-GUID: correctly, there is no meaningful check to design — **the GUID itself is the bearer credential**. Any client presenting a given GUID is treated as that profile; nothing in this scheme validates that the GUID's presenter is its original creator (no signature, no password, no server-issued rotating secret). This is explicitly friends-server-grade trust, not anti-cheat or account security: a malicious peer who reads another player's `NCMP-identity.json` (plaintext, local file, no OS ACL beyond normal file permissions) can impersonate them. This matches the existing security posture of the codebase (protocol has no TLS/auth-secret exchange beyond the schema-hash version check), so it's a consistent-with-status-quo weakening, not a new hole — but must be stated honestly rather than silently glossed as "secure."
