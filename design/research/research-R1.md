# Q1 â€” How the mod spawns dynamic entities today

**Player puppet recipe** (`code/assets/redscript/World/NetworkWorldSystem.reds:54-72`):
```
DynamicEntitySpec: recordID = t"Character.MaMuppet" | t"Character.WaMuppet" (chosen by ccstate isBodyGenderMale)
                   alwaysSpawned = true; persistState = false; persistSpawn = false
                   position/orientation from packet; tags = [n"CyberpunkMP.Puppet"]
CreateEntity via GameInstance.GetDynamicEntitySystem().CreateEntity(npcSpec) -> EntityID
```
"Ours" = the tag `n"CyberpunkMP.Puppet"` (matched in `AppearanceSystem.reds:61`, `InterpolationSystem.cpp:255`) plus flecs bookkeeping in `NetworkWorldSystem` (server-idâ†’flecs entity). Native `Spawn` (`NetworkWorldSystem.cpp:34-61`) calls the reds `CreatePuppet` by reflected function ptr (`.cpp:412`), then `AppearanceSystem::AddEntity` seeds equipment+ccstate+name, then `make_alive(serverId).emplace<SpawningComponent>(id)`.

**Vehicle recipe** (`World/VehicleSystem.reds:24-38`): identical spec shape, `recordID = <TweakDBID vehicle>`, `tags = [n"CyberpunkMP.Vehicle"]`, `alwaysSpawned = true`, no persistence.

**Workspot anim entity** (`World/AppearanceSystem.reds:181-217`, `HandleEmote`): spec uses `templatePath = r"mods\cyberpunkmp\workspot_anim.ent"` (not a recordID), `alwaysSpawned = true`, tag `n"CyberpunkMP.Puppet.Animation"`, positioned at the target puppet. On attach (`OnAnimationEntityAttached:220-232`) it calls `WorkspotGameSystem.PlayInDeviceSimple(animEntity, puppet, false, n"workspot_base", ...)` then `SendJumpToAnimEnt(entity, animName, true)`, and a 2s `DelayCallback` tears it down (`StopInDevice` + `DeleteEntity`). This is a proven emote/idle-animation attach path already in-tree.

**Critical AI fact:** the muppet records (`code/assets/Tweaks/CyberpunkMP.tweak:68-111`) derive from `Character.Panam` but override `reactionPreset = "ReactionPresets.NoReaction"`. That is the existing "AI-off pedestrian" precedent â€” remote puppets don't react/fight/wander because reactions are neutered at the record level and locomotion is hijacked externally (below).

# Q2 â€” Can the same machinery spawn civilian NPC records? What AI runs, how to disable/command it?

Yes. `DynamicEntitySystem.CreateEntity(spec)` accepts any `Character.*` recordID (that's exactly what player puppets use). Nothing structurally blocks `spec.recordID = t"Character.<civilian>"`. Constraints found:

- **Client spawn path is hardcoded to muppets.** `CreatePuppet(NetworkWorldSystem.reds:54)` only picks `MaMuppet`/`WaMuppet`; the wire `NotifyCharacterLoad` (`code/protocol/server.proto:22-30`) carries **no recordID/template field** (only id/pos/rot/is_player/equipment/ccstate/name). Any civilian-variety spawn needs a new protocol field + a `CreatePuppet` variant. Reusing the muppet record for pedestrians works with zero protocol change but every ped is a V-body clone.
- **AI available if you DON'T set NoReaction.** `AIComponent` (DUMP:`core/components/aiComponent.script`) exposes `SendCommand(AICommand):Bool`, `StartExecutingCommand`, `CancelCommandById`, `SetAIRole`, `GetHighLevelState`. Retrieved via `ScriptedPuppet.GetAIControllerComponent()`/`AIHumanComponent.Get(puppet, out)` (used throughout DUMP:`cyberpunk/NPC/NPCPuppet.script:3389+`).
- **Move commands exist** (DUMP:`core/ai/aiCommand.script`): `AIMoveToCommand extends AIMoveCommand` with `movementTarget:AIPositionSpec`, `movementType:moveMovementType`, `ignoreNavigation`, `desiredDistanceFromTarget`, `finishWhenDestinationReached`, `facingTarget`; also `AIMoveOnSplineCommand` (spline NodeRef). `AIPositionSpec.SetWorldPosition(...)` sets the target. So a walking ped = spawn record with reactions on + `aiComponent.SendCommand(moveToCmd)`.
- **Navmesh queries exist but are client-only** (DUMP:`core/systems/navigationSystem.script`): `NavigationSystem.CalculatePathOnlyHumanNavmesh(start,end,NavGenAgentSize.Human,tol)` -> `NavigationPath{array<Vector4>}`, `FindPointInSphereOnlyHumanNavmesh`, `FindNavmeshPointAwayFromReferencePoint`, `AddObstacle`. **These run inside the game (need streamed navmesh); the C++ server has no navmesh** â€” confirmed by absence of any nav code server-side and by these being an `IScriptable` game system reached via `GameInstance`.
- **Crowd system is essentially closed** (DUMP:`core/components/crowdMemberComponent.script`): `CrowdMemberComponent` is `importonly` with only high-level pokes (`ChangeMoveType`, `TryChangeMovementDirection`, `AllowWorkspotsUsage`). Real ambient crowd/traffic is engine-driven by the community/populate systems off world blockout data â€” **not authorable per-entity from redscript**, so vanilla crowd cannot be "captured and replicated." You must spawn your own puppets.
- **Disabling AI:** two proven levers â€” (a) record-level `reactionPreset = ReactionPresets.NoReaction` (already used); (b) simply never sending an AICommand and driving transform externally (current puppet approach). `AIComponent.DisableCollider()`/`EnableCollider()` exist for physics toggling.

# Q3 â€” Can external MovementController driving be applied to an NPC puppet (AI off, stream positions)?

**Yes â€” this is already how remote *players* work, and remote players are Panam-derived NPC puppets.** The driver: `InterpolationSystem.cpp` hooks `IdleController_SetAnimation` (`.cpp:220`), and for any entity tagged `CyberpunkMP.Puppet` (`.cpp:255`) it swaps in a `MultiMovementController` (`.cpp:266 AttachController`) and thereafter sets transform every frame via `aEntityComponent.Controller->SetTransform(pos, direction, speed)` (`.cpp:129`) fed by interpolated `NotifyEntityMove` timepoints (`.cpp:186-215`). A civilian puppet carrying the same tag rides this identical path with **zero client-code change** â€” it only needs a server feeding `NotifyEntityMove`. T1/T2/T3 differ only in *what generates those positions*.

# Q4 â€” Vehicles without a driver: can a dynamic vehicle be spawned and driven externally?

**Partially, and this is the weakest tier.** Spawn is trivial (`VehicleSystem.reds:24`). For remote-player vehicles the motion path is: driver-client reads its own `vehicle->rigidBody->currentTransform` and streams `MoveEntityRequest` (`NetworkWorldSystem.cpp:268-320`), and each *receiving* client applies it through a bespoke physics-forcing routine (`InterpolationSystem.cpp:81-118`): `SetSimpleMovement(...true)`, `EnablePhysics(vehicle,0x100)`, then `ForceMoveTo(vehicle, transform, dt)` (RawFunc hash `2933986803`). There is **no self-propelled/driverless vehicle path** â€” a driverless car needs a source of transforms. The code is riddled with "not working / crashes sometimes" comments around mounting (`VehicleSystem.reds:105-139`), and the engine-toggle offsets already drifted once on 2.31a (`VehicleSystem.cpp:253-258`). Driverless traffic = highest RE/stability risk.

# Which side runs the AI brain? (authority model from Level.cpp)

**What the repo supports today:** every replicated actor is `child_of(a player)` (`Level.cpp:191-194, 314`). Movement writes are authorized **only** when `target.parent()` has a `PlayerComponent` whose `Connection == sender` (`Level.cpp:212-230`). Replication broadcasts `NotifyEntityMove`/`NotifyCharacterLoad` to everyone **except the owning parent** (`MovementComponent.cpp:37`, `Level.cpp:86,113`). There is **no server-simulated, ownerless entity anywhere today**, and the C# plugin SDK is **read-only**: `MovementComponent.cs` exposes only getters, and there is **no spawn/CreateEntity API in C#** (grep of `code/server/scripting` returns none) â€” only chat/RPC/timers/read-position (`Player.cs`, `World.cs` is just an `UpdateEvent`).

Two viable ownership models:

- **A. Server-side simple sim (recommended for T1/T2-waypoint).** Create an ownerless flecs entity natively and `set<MovementComponent>` each tick â€” the existing `OnSet` observer (`MovementComponent.cpp:57-66`) auto-replicates to **all** clients (owner check never matches a null parent â†’ true "same NPC on every PC"), and clients render it through the unchanged `HandleCharacterLoad`â†’`CreatePuppet` path. **Requires new native C++ + new SDK bindings** (C# can't spawn/write today). Limited to scripted/waypoint motion because the **server has no navmesh**.

- **B. Delegate authority to one client (needed for navmesh-correct T2 and any T3).** Parent the NPC to a chosen player (`child_of(player)`); that client runs the real AI brain â€” `AIComponent.SendCommand(AIMoveToCommand)` and `NavigationSystem` path queries, both client-only â€” and streams `MoveEntityRequest` exactly like its own puppet (`NetworkWorldSystem.cpp:250` `UpdatePlayerLocation`). The existing authority check passes because `parent == that player`. Costs: the NPC only lives while that client has it streamed in; you need **hand-off on disconnect/range**; and it opens a **client-authoritative cheat surface** (that client can teleport the NPC), which the current model already tolerates for player puppets.

**Recommendation:** T1 and waypoint-T2 â†’ model A (server-authoritative, cheat-proof, cheapest to keep consistent). Navmesh-accurate T2 and any T3 â†’ model B (only the client has navmesh + vehicle physics), accepting the hand-off/cheat tradeoffs.

**UNVERIFIED:** whether a civilian record spawned *with* reactions enabled ticks combat/wander AI that conflicts with external transform driving â€” not testable without a compile+in-game run; the safe, proven configuration is the shipped `ReactionPresets.NoReaction` + external drive.
