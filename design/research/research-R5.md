# R5 — Server-side persistence & player profiles

## Q1. Persistence reality today

- **WorldClock** (`code/server/scripting/JobSystem/WorldClock.cs:19-126`): the only working persistence in the whole server. `Load()` (:95-111) reads `world-clock.txt`, `Save()` (:113-125) writes it, from `File.Exists`/`File.ReadAllText`/`File.WriteAllText` against a **bare relative path** — i.e. whatever the C# host process's current working directory is at runtime (not resolved via any helper). Called on a 300s-ish timer (`Server.Timer.Add(Broadcast, 60.0f)` → `Broadcast` calls `Save()`, :51,71) and never on shutdown explicitly.
- **JobManager.Save() is a stub**: `code/server/scripting/JobSystem/JobManager.cs:128-131` — logs `"Saving Jobs..."` and does nothing. `OnPlayerJoin` (:114-117) is `// Load from db` — a comment, no code. Confirms zero per-player persistence exists anywhere in the C# plugin layer today.
- Grep of all server C#/C++ for `File.`/`sqlite`/`litedb`/`.db`/`Directory.` (`grep -rniE "sqlite|litedb|\.db\b|File\.(Write|Read|Open|Append|Exists|Create)|StreamWriter|StreamReader|JsonSerializer|Newtonsoft" code/server`) hits only: `Server.cs` (loading plugin DLLs from disk), `Plugins.cs` (plugin package download/extract/config-file scaffolding, `code/server/loader/Systems/Plugins.cs:73-330`), `SdkGenerator/Program.cs` (codegen, irrelevant), and `WorldClock.cs`. No sqlite/litedb/db references anywhere in the repo.
- **Native side** (`code/server/native`) has exactly one persistence point: `GameServer::GameServer()` reads/creates `config/server.json` via `nlohmann::json` (`code/server/native/GameServer.cpp:41-58`), using `std::ifstream`/`std::ofstream` directly (not the `LoadFile`/`SaveFile` helpers). The `Config`/`FlecsConfig` structs use `NLOHMANN_DEFINE_TYPE_INTRUSIVE` for struct↔JSON (`code/server/native/Config.h:15,48`) — this is the established idiom for any new persisted struct.
- **Working-directory convention**: `GetPath()` (`code/common/Core/Filesystem.h:8`, impl `code/common/Core/Filesystem.cpp:13-26`) returns the loaded module's directory on Windows (`GetModuleFileNameW` against `__ImageBase`) and `std::filesystem::current_path()` everywhere else (i.e. non-Windows native server = **process CWD**, same ambiguity as the C# side). `SaveFile`/`LoadFile` (`Filesystem.cpp:28-50`) are raw-bytes helpers, already available but unused by `GameServer.cpp`'s config code (which reaches straight for `std::ifstream`/`std::ofstream`).
- Net effect: **two inconsistent conventions coexist** — native puts state under `GetPath()/"config"/...` (subdirectory next to the binary/CWD), the C# JobSystem plugin writes a bare filename directly into CWD with no subdirectory and no use of `GetPath()`-equivalent resolution. Any new persistence should pick one; recommend following the native `config/`-sibling-subdirectory pattern (e.g. `players/`) since it scales to many files, over WorldClock's bare-file precedent.

## Q2. Profile store design — recommendation: **JSON-per-player files, native C++, `nlohmann::json`**

Do **not** reach for `Microsoft.Data.Sqlite` or LiteDB for v1:
- Checked all 6 `.csproj` in `code/server` (`Server.Loader`, `CyberpunkSdk`, `EmoteSystem`, `JobSystem`, `SdkGenerator`, `VehicleSystem`) — zero `PackageReference` to Sqlite or LiteDB anywhere (only refs present: `CppSharp.Runtime`, `CurseForge.APIClient`, `EmbedIO`, `LibGit2Sharp`).
- `~/.nuget/packages` contains **no** sqlite/litedb packages (`find ~/.nuget -iname "*sqlite*" -o -iname "*litedb*"` → empty). So neither is currently vendored/cached offline; picking either means a first build needs a live NuGet restore. Whether this box has that network access at build time is **UNVERIFIED** — orthogonal to the repo, not something grep can settle — but the absence of any prior use of either package in this codebase is itself evidence they're not the established path.
- More decisively (see Q4): the data that actually needs persisting (ccstate/equipment) is produced and consumed entirely in **native C++** (`Level.cpp`), which the C# plugin layer cannot currently see at all. Choosing a C#-only DB engine doesn't fit where the write/read actually has to happen.

**Recommendation**: extend the existing native JSON convention. Add a `PlayerProfile` struct with `NLOHMANN_DEFINE_TYPE_INTRUSIVE` (same macro already used by `Config`, `code/server/native/Config.h:15,48`), one file per player, using the already-linked `nlohmann::json` (`GameServer.cpp:16`) and the existing `LoadFile`/`SaveFile` helpers (`code/common/Core/Filesystem.h/.cpp`) instead of raw streams.

**File layout**: `GetPath()/"players"/<username>.json` (mirrors `GetPath()/"config"/"server.json"`, `GameServer.cpp:43,48,50`). Username is the only identity signal that exists today (see R6 note below), so it doubles as filename key; sanitize before use as a path component (not done anywhere yet — new code).

**Schema** (fields required by the task, typed against what already exists on the wire):
```cpp
struct PlayerProfile {
    std::string username;            // key; == PlayerComponent::Username (PlayerComponent.h:7)
    std::vector<std::string> equipment; // mirrors AppearanceComponent::equipment (AppearanceComponent.h:4)
    std::vector<uint8_t> ccstate;       // mirrors AppearanceComponent::ccstate (AppearanceComponent.h:5)
    std::string bodyTemplate;           // e.g. "Character.MaMuppet"/"Character.WaMuppet" choice, not tracked anywhere today — new field
    float posX, posY, posZ;             // position-at-logout; glm::vec3 has no existing json adapter (grep confirmed none in code/server or code/common) — store as scalars or add one
    float rotX, rotY, rotZ;
    int64_t createdAtUnix;
    int64_t lastSeenUnix;
    NLOHMANN_DEFINE_TYPE_INTRUSIVE(PlayerProfile, username, equipment, ccstate, bodyTemplate,
        posX, posY, posZ, rotX, rotY, rotZ, createdAtUnix, lastSeenUnix)
};
```
`MovementComponent` (`code/server/native/Components/MovementComponent.h`) already has exactly `Position`/`Rotation` as `glm::vec3` plus `Velocity`/`Tick` — the profile's position-at-logout fields should mirror that shape once a `glm::vec3`↔json adapter is added (none exists in-repo today, confirmed by grep).

## Q3. Server-authoritative ccstate — flow inversion design

**Today** (confirmed by reading, all in `code/server/native/Game/Level.cpp:164-200`):
1. Client sends `client.SpawnCharacterRequest{position, rotation, equipment, ccstate}` (`code/protocol/client.proto:14-21`) built from whatever the local `CharacterCustomizationSystem` populated (`code/client/App/World/NetworkWorldSystem.cpp` comments at :15-20 re: `kSpawnCcstateTimeout`).
2. `Level::HandleSpawnCharacterRequest` (`Level.cpp:164-200`) unconditionally trusts the client payload, builds `pComponent->Puppet = ... .set<AppearanceComponent>({equipment, ccstate})` (`Level.cpp:191-194`), and acks with `server.SpawnCharacterResponse{id, cookie}` (`server.proto:17-20`) — **no appearance data goes back to the sender**, only the entity id.
3. `Level::Add(player)` (`Level.cpp:199`) → `AddPlayer` (`Level.cpp:139-157`) → generic `Add()` on the puppet (`Level.cpp:60-93`) broadcasts `server.NotifyCharacterLoad{position, rotation, equipment, ccstate, name}` (`server.proto:22-30`, built by `Level::Serialize`, `Level.cpp:359-394`) to every **other** connected player (`Level.cpp:84-90`, excludes `owner`) — this is how remote puppets get their appearance client-side, via `NetworkWorldSystem::HandleCharacterLoad` (`code/client/App/World/NetworkWorldSystem.cpp:183-223`) → `AppearanceSystem::AddEntity` (:53) storing per-`EntityID` ccstate/equipment/name, consumed later by `AppearanceSystem::ApplyAppearance` (`code/client/App/World/AppearanceSystem.cpp:171-...`) at redscript's `OnEntityAssemble` hook (`code/assets/redscript/World/AppearanceSystem.reds:238-239`). The owner never receives this message about themself (`owner == aEntity` is excluded, `Level.cpp:86`).
4. The owning client's own on-screen character comes from the game's normal local save/character-creator flow, entirely independent of the network layer — the network puppet is only what *other players see*.

**Minimal inversion for returning players**:
1. **Where to intercept**: `Level::HandleSpawnCharacterRequest` (`Level.cpp:164-200`) is the one place `AppearanceComponent` gets constructed — this is the only seam that matters. Before `.set<AppearanceComponent>({equipment, ccstate})` (:194), look up the player entity's `PlayerProfile` (attached at auth time, see Q4). If a stored profile exists and its `ccstate` is non-empty: use `profile.equipment`/`profile.ccstate` in place of `aMessage.get_equipment()`/`get_ccstate()`, and use `profile.pos*/rot*` in place of `aMessage.get_position()`/`get_rotation()` (:168-169) instead of the client-submitted spawn point. If no profile / empty stored ccstate (new player): keep current behavior, and additionally persist `equipment`/`ccstate` into a freshly-created `PlayerProfile` right here (first-time write).
2. **Telling the owner "you exist, here is your char"**: this is the genuinely new wire message, because today nothing server→self carries appearance. Two options, evaluated:
   - (a) Bloat `SpawnCharacterResponse` (`server.proto:17-20`) with optional `equipment`/`ccstate`/`position`/`rotation` fields. Rejected: `HandleSpawnCharacterResponse` client-side (`NetworkWorldSystem.cpp:230-239`) currently does one thing only — `SetRemotePlayerId(aMessage.get_id())` — and conflating the connection-ack semantics with appearance payload muddies a message other code already depends on.
   - **(b) Recommended — new message, reusing the existing shape**: add `NotifyOwnCharacterLoad` to `server.proto`, field-for-field identical to `NotifyCharacterLoad` (`server.proto:22-30`: position, rotation, is_player, equipment, ccstate, name), registered as `Protocol.content` field `13` (next free tag after `notify_vehicle_control_assigned = 12`, `server.proto:101`). Server sends it **only to the owning connection** (`GServer->Send(aMessage.ConnectionId, ...)`), immediately after resolving the profile in `HandleSpawnCharacterRequest`, but **only when returning-player data was actually substituted** (skip entirely for new players — no-op preserves current behavior/wire traffic for the common case).
   - **Client-side handling** (new, does not exist): a handler parallel to `HandleCharacterLoad` (`NetworkWorldSystem.cpp:183-223`), but instead of `Spawn(...)` (which creates a *new* dynamic puppet via `CreatePuppet`/`DynamicEntitySpec` per the R context), it must apply appearance to the **already-existing local player object**: call `AppearanceSystem::AddEntity(localPlayerEntityId, equipment, ccstate, name)` then explicitly invoke `AppearanceSystem::ApplyAppearance(localPlayerObject)` (`AppearanceSystem.cpp:171`, confirmed generic — it looks up by `object.instance->entityID` in `m_playerCcstate`, does not care whether the object is a dynamically-spawned puppet or the live player, `AppearanceSystem.cpp:173-179`) — i.e. the exact same primitive already used for remote puppets, just driven manually instead of via the `OnEntityAssemble` redscript hook (`AppearanceSystem.reds:238-239`), since the local player object is never "assembled" through that path. Position/rotation would need a separate local teleport call (no existing client-side "teleport self" entry point was found in this pass — flagged as a gap, not fabricated).

## Q4. Where "has this player been here before" gets answered: **native, not C#**

The C# `CyberpunkSdk` plugin layer's visibility is limited to what `PlayerSystem`/`Player` expose (`code/server/scripting/CyberpunkSdk/Game/PlayerManager.cs:1-25`, `Game/Player.cs:1-59`):
- `PlayerJoinEvent`/`PlayerLeftEvent` carry only a raw `ulong Id` (`PlayerManager.cs:7-10`) — fired from native `PlayerManagerScriptInstance` off a flecs `OnSet`/`OnRemove` observer on `PlayerComponent` (`code/server/native/Scripting/PlayerManagerScriptInstance.cpp:53-69`), i.e. **after** `PlayerManager::Create` already ran (`code/server/native/PlayerManager.cpp:27-44`, called from `GameServer::HandleAuthentication` at `GameServer.cpp:212`).
- `Player.Username` (`Player.cs:54-57`) is available at that point — confirmed the native `PlayerComponent::Username` is set synchronously in `PlayerManager::Create` (`PlayerManager.cpp:34`) before the observer fires (:36) — so **the C# layer can answer "have I seen this username" today**, just from `PlayerJoinEvent` + `Server.PlayerSystem.GetById(id).Username`.
- But grepping `code/server/scripting/CyberpunkSdk` for `Spawn`/`Puppet`/`Appearance`/`ccstate` returns **only `Player.cs`'s own `PuppetId` getter** — there is no SDK event, hook, or property anywhere exposing `SpawnCharacterRequest`, `AppearanceComponent`, or ccstate/equipment to C#. That whole flow is native-only (`Level.cpp`), with zero script-instance surface (confirmed via `find code/server/native/Scripting -iname "*.h" | xargs grep -l "Spawn\|Appearance"` → no matches).

**Conclusion**: the boolean "seen before" check *could* be answered from either layer (username is visible to both by join time), but the **action** that matters — substituting stored ccstate/equipment/position into `AppearanceComponent` — can only happen where `AppearanceComponent` is built, which is exclusively native (`Level.cpp:191-194`). Building this in C# for v1 would require adding brand-new native→script plumbing (a `SpawnCharacterRequest` hook + a script-writable override channel back into `Level::HandleSpawnCharacterRequest`) that doesn't exist today — pure overhead for no benefit. **Recommend doing all of R5 (profile lookup, ccstate override, persistence) natively**, attaching a `PlayerProfile` (loaded/created) as a new flecs component on the player entity inside `GameServer::HandleAuthentication` (`GameServer.cpp:193`, right after the username is logged and before `PlayerManager::Create` at :212, or as a follow-up `.set<PlayerProfile>(...)` on the entity `Create` returns), then reading it back in `Level::HandleSpawnCharacterRequest` (`Level.cpp:166` already resolves `player` via `GetByConnectionId`).
