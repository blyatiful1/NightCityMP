# R4 — World-Save Transfer: Findings

## Q1 — Server HTTP surface

**What exists (EmbedIO, C#, in `Server.Loader`):**
- `code/server/loader/Systems/WebApi.cs:42-56` (`CreateWebServer`) builds one `WebServer` with these registered routes/modules, in this exact order:
  1. `ActionModule("/api/v1/mods/", GET)` → `HandleModsRoute` (`WebApi.cs:124-130`)
  2. `ActionModule("/api/v1/statistics/", GET)` → `HandleStatistics` (`WebApi.cs:132-138`)
  3. `RegisterAuthentication(server)` (`WebApi.cs:95-120`) — installs `AdminAuthModule` (`WebApi.cs:156-168`), a `BasicAuthenticationModule` on base route `"/"`. **Skipped entirely in Debug** (`EnvironmentHelper.IsDebug`), so no HTTP auth exists in dev builds. In Release it requires env vars `CYBERPUNKMP_ADMIN_USERNAME`/`CYBERPUNKMP_ADMIN_PASSWORD` (`WebApi.cs:103-109`, throws `SecurityException` if unset) and whitelists only `/api/v1/mods` and `/api/v1/statistics` from the credential check (`WebApi.cs:113-116`).
  4. `RegisterPlugins(server)` (`WebApi.cs:58-83`) — per-plugin static asset folders, per-plugin `WebApiController`s, and `/api/v1/plugins/` list.
  5. `RegisterAssets(server)` (`WebApi.cs:85-93`) — catch-all static folder at `/` serving `FileSystemHelper.GetPath("assets")`.

  Because modules 4 and 5 register **after** the auth module and are not in its whitelist, they require admin Basic-Auth in Release. A save-download route reachable by ordinary players must either be registered **before** `RegisterAuthentication(server)` (mirroring the mods/statistics pattern) or added to the `AdminAuthModule` whitelist array — admin credentials are not something a joining player should need.

- **Port wiring bug/inconsistency**: `WebApi.cs:31` binds to `IConfig.Get().Port`, which — per the C++→C# binding (`code/server/scripting/CyberpunkSdk/CyberpunkSdk.Internal.cs:1219,1341-1348`) — maps to `Config::Port` (default **11778**, `code/server/native/Config.h:27`), i.e. the **same port number as the GameNetworkingSockets game-protocol listener** (`code/server/native/GameServer.cpp:67-72`). `Config::WebPort` (default **11779**, `Config.h:28`) exists but is **never read anywhere in the C# loader** (only hit in `select:` grep as the struct field / native accessor). This isn't a runtime conflict (GNS is UDP, `HttpListener`/EmbedIO is TCP — independent OS port namespaces), but it means an admin setting `WebPort` in `server.json` has zero effect today; the HTTP API is actually reachable at `http://<ip>:11778/...`, not `:11779`. Any new save-serving route inherits this same live port until someone fixes the `Port`→`WebPort` mismatch.

- **TCP "admin API on :11778"** in the task prompt is actually the game protocol port (GNS), not a separate admin API — confirmed by `Settings.h:25` (`port = 11778`) and `code/assets/redscript/NightCityMP/MainMenu.reds:218` (`let port: Uint32 = 11778u;`, the JOIN address-parsing default). There is no separate TCP admin protocol; "admin" surface = the EmbedIO HTTP server above.

**Where a `GET /world-save` route would go**: add an `ActionModule("/api/v1/world-save/", HttpVerbs.Get, HandleWorldSaveRoute)` (and a `HEAD` variant, or a small JSON metadata route `/api/v1/world-save/meta`) into the `.WithModule(...)` chain in `WebApi.cs:46-50`, alongside mods/statistics, plus a `HandleWorldSaveRoute` method in the `#region Routes` block (`WebApi.cs:122-153`). No server-side "canonical sav.dat" concept exists anywhere yet — confirmed by `grep -rl "sav.dat|WorldSave"` across the repo, which hits **only** `code/client/App/Saves/SaveManager.cpp` (a client-side file-existence check). The server has no analog; it must be built from scratch, most naturally next to `GameServer::m_path` (`code/server/native/GameServer.h:63`, same directory `GameServer.cpp:41-43` already uses for `config/server.json`), following the exact persistence precedent of `code/server/scripting/JobSystem/WorldClock.cs:1-19` (a plain sidecar state file next to the server binary, not touching game-authoritative files).

## Q2 — Client download ability

**Client deps (`code/client/xmake.lua:1,37-56`)**: `mimalloc, spdlog, hopscotch-map, cryptopp, gamenetworkingsockets, imgui, minhook, mem, openssl, zlib, xbyak, glm, nlohmann_json, entt, wil, nameof, semver, microsoft-gsl, flecs`. **No HTTP client library** (no curl, cpr, httplib, WinHTTP wrapper) anywhere in this list or in `add_syslinks(...)` (`xmake.lua:19-33`, all raw Win32 libs: `user32, shell32, ole32, bcrypt, dxgi, d3d12, ...` — no `winhttp`/`wininet`).

**Game protocol transport**: `gamenetworkingsockets` (Valve GameNetworkingSockets/GNS), confirmed by `code/common/Network/Client.cpp` and `Server.cpp` (`ISteamNetworkingSockets`, `SteamNetworkingIPAddr`, `SendMessageToConnection`, snappy-compressed payloads with a 1-byte opcode header: `kPayload/kCompressedPayload/kHandshake/kServerTime` — `Server.cpp:243-265`, `Client.cpp:314-342`). This is UDP-based, message-oriented, no existing app-level chunking/fragmentation logic anywhere in `code/common`.

**Hard size ceiling for the GNS route** (verified locally, not web-sourced): the vendored GameNetworkingSockets 1.6.0 headers at `/home/crocco/.xmake/packages/g/gamenetworkingsockets/v1.6.0/*/include/steam/steamnetworkingtypes.h:846` define `k_cbMaxSteamNetworkingSocketsMessageSizeSend = 512 * 1024` (512 KiB) — a hard per-`SendMessageToConnection` cap. A real ~2 MB `sav.dat` (measured 1.98–2.12 MB on two local saves, see Q4) would need **≥4 manual chunks** through the protobuf `oneof Protocol` (`code/protocol/client.proto`, `server.proto`) if sent over the game protocol — the current design has zero infrastructure for that (no chunk/reassembly messages exist in either `.proto`).

**Recommendation — HTTP over chunked-protocol**: Given (a) the size cap above, (b) EmbedIO already runs an HTTP surface server-side (Q1), and (c) the client is a Windows-only PE target (`add_ldflags("/LARGEADDRESSAWARE")`, `xmake.lua:9`) that already calls raw Win32 APIs directly (`SHGetKnownFolderPath`, `ShlObj.h` in `SaveManager.cpp:3,64-75`) rather than wrapping them, the lowest-risk option is **native WinHTTP** (`winhttp.dll`, ships with Windows/Proton's Windows layer) via `add_syslinks("winhttp")` in `xmake.lua` — zero new third-party dependency, consistent with the existing "call the Win32 API directly" style. **cpp-httplib** is a reasonable header-only fallback if WinHTTP's C API is judged too verbose, but it adds a vendor dependency for something the OS already provides. Extending the game protocol with chunked blob messages is not recommended: it duplicates transport-layer work GNS already can't do cheaply at this size, and reliable GNS messages already pay a reassembly cost internally — stacking a second reassembly layer on top for a one-shot 2 MB blob is unjustified complexity next to "just do an HTTP GET." **UNVERIFIED**: whether EmbedIO's built-in static-folder handler (`WebApi.cs:89-92`) actually serves `ETag`/`Range`/`If-Modified-Since` correctly was not confirmed from local source (EmbedIO ships as a prebuilt `.dll`, not source, in this tree) — recommend not depending on it and instead exposing an explicit JSON metadata route (hash/size/version) alongside the raw file route.

## Q3 — Install side: what's missing

**What exists** (`code/client/App/Saves/SaveManager.{h,cpp}`, native module + `code/assets/redscript/Saves/SaveManager.reds` bindings, all landed in commit `8eab363`):
- `SavesDir()` (`SaveManager.cpp:62-75`) — `FOLDERID_SavedGames` resolution.
- `ListSaves`/`ListMpSaves` (`SaveManager.cpp:244-252`) — enumeration + `metadata.9.json` read (`ReadMetadata`, `SaveManager.cpp:80-101`).
- `CreateSaveFromTemplate` (`SaveManager.cpp:266-316`) — copies a **template folder already on disk** (`kSeedSubdir = "assets/Saves"`, `SaveManager.cpp:34`) into `NCMP-<name>`, then `TryRewriteMetadataName` (`SaveManager.cpp:206-231`) patches `metadata.9.json`'s `Data.metadata.name` field to match.
- `SetPendingSave`/`PendingSaveAllowsConnect` (`SaveManager.cpp:318-336`) — the auto-connect gate consumed by `code/assets/redscript/World/NetworkAutoConnect.reds:38-44`.

**What is missing, concretely:**
1. **No function installs a downloaded blob.** `CreateSaveFromTemplate` only does `fs::copy` from an existing template folder (`SaveManager.cpp:306`); there is no `InstallDownloadedSave(serverId, savDatBytes, metadataBytes)`-shaped native that writes an in-memory buffer to `SavesDir() / ("NCMP-" + serverId) / {sav.dat, metadata.9.json}`. This needs to be added, following `InstallSeedIfAbsent`'s (`SaveManager.cpp:174-199`) directory-creation/overwrite pattern and `TryRewriteMetadataName`'s (`SaveManager.cpp:206-231`) metadata-patch pattern, but writing bytes instead of copying a folder. A real save folder is exactly 3 files (verified locally: `sav.dat` ~2.0–2.1 MB, `metadata.9.json` ~2.7 KB, `screenshot.png` ~110–130 KB — e.g. `.../Saved Games/CD Projekt Red/Cyberpunk 2077/AutoSave-14/`); `screenshot.png` is cosmetic and can be omitted/blanked in the transfer to shave ~100 KB.
2. **No "server-id" concept exists anywhere.** `Config` (`code/server/native/Config.h:18-48`) has `Name`, `Description`, `ApiKey` (used only for a Cloudflare-style `ApiClient`, `code/server/loader/Systems/Plugins.cs:31,119` — unrelated to save identity), but nothing that functions as a stable per-server ID for the `NCMP-<server-id>` folder-naming scheme the task assumes. This must be added (e.g. a GUID generated once and persisted into `config/server.json` the same way `GameServer`'s constructor already round-trips that file, `GameServer.cpp:41-58`), and exposed to the client (naturally via the same metadata endpoint from Q1/Q2, since the client needs it *before* it has any other server info).
3. **The JOIN-flow integration seam is an explicit, unfinished stub — not just "missing," but documented as deliberately deferred.** `code/assets/redscript/NightCityMP/MainMenu.reds:239-262` (`RouteToMpSaveSelection`) currently just does `this.m_menuEventDispatcher.SpawnEvent(n"OnLoadGame")` — the vanilla, **unfiltered** Load Game screen — with a comment explicitly stating the `Saves.MpSaveManager` calls are "intentionally NOT referenced here" pending a P3 integration step (`MainMenu.reds:250-262`). Confirmed via `git log`: `MainMenu.reds`'s latest touch (`01f3cba`) postdates `SaveManager`'s introduction (`8eab363`, ancestor-checked), so this is not stale documentation — the wiring genuinely has not been done for *any* MP save yet, let alone a server-downloaded one. This is the single seam where a download-then-install-then-resolve-index flow needs to be inserted.
4. **Index resolution for `LoadSavedGame`** — confirmed from the CDPR source dump (`/home/crocco/Games/Heroic/.../tools/redmod/scripts/core/ui/systemRequestsHandler.script:38` `LoadSavedGame(saveId: Int32)`; `cyberpunk/UI/fullscreen/pregame/loadGameMenu.script:436-440` `OnSavesForLoadReady(saves: array<String>)`): the save index is simply the **array position** in the string list `RequestSavesForLoad()` → `OnSavesForLoadReady` returns — matching `SaveManager.reds:15-16`'s own comment that "saveIndex is not stable," i.e. it must be re-resolved fresh every time by linear-searching that array for the just-installed folder's internal name (`"NCMP-<server-id>"`) and using its position. No code currently does this search+load sequence anywhere in the tree (`RouteToMpSaveSelection` never calls `RequestSavesForLoad`/registers `OnSavesForLoadReady` at all — it just delegates to the vanilla menu).

## Q4 — Versioning / design tension

**Where a hash/version signal would flow**: `server::Settings` (`code/protocol/server.proto:8-10`) currently carries only `uint32 update_rate = 1`, populated in `GameServer::HandleAuthentication` at `code/server/native/GameServer.cpp:195-197` and consumed client-side at `NetworkService::HandleAuthentication` (`code/client/App/Network/NetworkService.cpp:97-125`, `m_settings = aResponse.get_settings()`). This is a real, already-wired extension point — **but it fires too late**. Per `code/assets/redscript/World/NetworkAutoConnect.reds:33-44`, the GNS authentication handshake only happens **after** a save has already been loaded into the world (`OnGameAttached`, gated by `!IsPreGame()`). A first-time joiner has no `NCMP-<server-id>` save yet at that point — `LoadSavedGame` must already have succeeded before authentication ever occurs. So a hash-in-`AuthenticationResponse` design only tells you "your already-installed copy is stale, re-sync for *next* time" — it cannot drive the *first* download, which must happen earlier, in `RouteToMpSaveSelection` (Q3 point 3), decoupled from GNS entirely. **Recommendation**: do the version check and the download itself purely over HTTP against the Q1/Q2 endpoint (a `GET /api/v1/world-save/meta` returning `{hash, size, version, server_id}` compared against a small client-side sidecar file — not the game's own `metadata.9.json`, which has a CDPR-owned schema per `SaveManager.cpp:78-79`'s comment and shouldn't be extended with custom fields) at the JOIN/save-selection step, fully before `LoadSavedGame` is ever called. Keep the `Settings.world_save_hash`/`version` fields too (cheap to add) purely as a **post-load drift indicator** for logging/UX ("your world may be out of date, rejoin to resync"), not as the trigger mechanism.

**The design tension (flagged, per instructions)**: a CP2077 `sav.dat` is one opaque blob mixing **character state** (appearance, inventory, skills, personal quest flags) with **world state** (map exploration, world-quest flags, discovered locations) — there is no seam in the format the mod can exploit to sync only the world half. Server-authoritative "world" (`WorldClock`/weather precedent, `code/server/scripting/JobSystem/WorldClock.cs`) currently works precisely *because* it lives outside any save file (a plain `world-clock.txt` broadcast over RPC). Making `sav.dat` server-authoritative inverts that: every time the server pushes a newer save, a player's own local progress since their last sync (anything the save format can't separate out) is **overwritten**, because there's no per-field merge — only whole-file replace. Three options, none free:
- **(A) Full overwrite (simplest)**: server's `sav.dat` is the only truth; each player's `NCMP-<server-id>` local save is always replaced with the server's latest on version mismatch. Any character progress a player made is lost if the server's copy predates it (e.g. player played briefly before the server pushed an update). Matches the "world save," not "character save," framing in the task title, but is destructive to any local drift.
- **(B) One-way ratchet, no auto-overwrite of a *loaded* session**: only install the server's copy for players who don't have one yet (first join) or whose local copy predates the server's according to a monotonic version counter, and never touch a save that's ahead of or equal to the server's — punts the mixing problem to "don't update while someone might have unsynced local play," at the cost of divergence never actually healing.
- **(C) Split world/character state at the mod layer instead of the save layer**: keep `sav.dat` purely per-player/local (no server hosting at all), and extend the already-proven `WorldClock`-style server-authoritative sidecar-state model to whatever specific world facts actually need to be shared (quest flags, discovered map, NPC states) via new RPCs, applied on top of a vanilla per-player save at connect time. This avoids ever shipping a whole `sav.dat` over the wire and sidesteps the mixing problem entirely, at the cost of enumerating and separately implementing sync for each world fact (much larger scope than "download a file").
None of these is implemented today; this is a decision the task owner needs to make explicitly before Q1–Q3's plumbing is worth building, since it changes what "world save" even means. UNVERIFIED: whether CP2077's save format has any documented internal separation between character and world sections that could support (C)'s "split" partially at the save level too — not investigated (would require reverse-engineering `sav.dat`'s binary/serialized layout, out of scope for this pass).
